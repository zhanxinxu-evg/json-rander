## 1. 系统简介

### 1.1 什么是 JSON Schema

在我们的系统中，AI 输出的图表数据是 JSON 格式。而 **JSON Schema** 是对这份 JSON 数据的"规格说明"——它定义了：

- 数据里应该有**哪些字段**
- 每个字段应该是**什么类型**（文本、数字、列表……）
- 哪些字段是**必填**的、哪些可选
- 每个字段的**取值约束**（范围、长度、格式等）

用日常的类比：如果 JSON 数据是一张**已经填好的表单**，那 JSON Schema 就是**空白表单的模板设计**——规定了每个格子叫什么、该填什么格式的内容。

例如，一张柱状图的 JSON 数据长这样：

```json
{
  "label": "各季度营收",
  "values": [
    { "label": "Q1", "value": 12.5 },
    { "label": "Q2", "value": 14.8 }
  ],
  "unit": "$B"
}
```

而你要设计的 JSON Schema 就是说明这份数据的规格：`label` 是必填文本、`values` 是对象列表、`unit` 是可选文本。具体的 JSON Schema 格式见 [3.7 节](#37-从表格到-json-schema)。

### 1.2 工作原理

AI 在回复中输出一段图表数据 JSON → 前端自动识别并渲染为可视化图表。

### 1.3 你需要做什么 / 不需要做什么

| 你的职责 | 不需要关心 |
|---|---|
| 明确图表用途——解决什么场景、展示什么信息 | 前端组件的实现代码 |
| 设计 Schema——定义数据字段、类型和含义 | 组件注册流程 |
| 编写 AI 使用说明——告诉 AI 何时用、怎么填 | 动画效果等技术细节 |
| 考虑边界情况——空值、极端数据的处理 | |

### 1.4 图表数据的整体结构

所有图表遵循同一个外层格式，你只需设计 `chartData` 内部的字段：

```json
{
  "root": "chart",
  "elements": {
    "chart": {
      "type": "你的图表类型名",
      "props": {
        "chartData": {
          "type": "你的图表类型名",
          "label": "图表标题",
          "...": "← 你设计的数据字段放这里"
        }
      },
      "children": []
    }
  }
}
```

- `type`：图表类型名称，PascalCase 格式（如 `RadarChart`）
- `label`：图表标题，必填

其余的 `root`、`elements`、`children` 等外层结构是固定的，不需要关心。

## 2. 完整示例：设计 RadarChart

> 以下是一个产品经理设计新图表的完整交付物。如果你是第一次设计图表，先通读此示例了解最终产出物的全貌，再根据需要深入后续的详细规则章节。

---

### 2.1 需求澄清

| 问题 | 回答 |
|---|---|
| 图表名称 | `RadarChart`（雷达图 / 蛛网图） |
| 一句话描述 | 在多个维度上对比一个或多个实体的能力分布，以雷达网格形式直观展示各维度强弱 |
| 典型场景 | AI 模型多维度能力对比、竞品产品特性对比、团队成员技能评估、投资组合风险维度分析 |
| 与现有图表的区别 | 见下表 |
| 最少数据点 | 3 个维度 |
| 建议上限 | 8 个维度、3 个实体 |

**与现有图表的区别**

| 对比图表 | 区别 |
|---|---|
| BenchmarkHeatmap | Heatmap 是表格式色块，适合实体较多；RadarChart 是图形化雷达网格，更直观但实体数量受限 |
| VersusChart | Versus 限定两个实体一对一对比；RadarChart 支持 1-3 个实体叠加 |
| BarChart | BarChart 是一维柱状对比；RadarChart 展示多维度的"形状"差异 |

### 2.2 Schema 定义

#### chartData 专属字段

| 字段 | 类型 | 必填 | 说明 | 约束 |
|---|---|---|---|---|
| `radarAxes` | `string[]` | 是 | 维度名称列表 | 3-8 个；每个名称不超过 10 字；不允许重复 |
| `radarEntities` | `RadarEntity[]` | 是 | 实体列表 | 1-3 个 |

#### 子类型：RadarEntity

| 字段 | 类型 | 必填 | 说明 | 约束 |
|---|---|---|---|---|
| `label` | `string` | 是 | 实体名称 | — |
| `values` | `number[]` | 是 | 各维度数值 | 长度必须等于 radarAxes 长度；建议 0-100 范围 |
| `color` | `string` | 否 | 自定义颜色 | HEX 格式；默认使用系统配色 |
| `highlight` | `boolean` | 否 | 是否高亮此实体 | 默认 false |

#### 数据量建议

| 参数 | 最小 | 推荐 | 最大 |
|---|---|---|---|
| 维度数 | 3 | 5-6 | 8 |
| 实体数 | 1 | 2 | 3 |

#### JSON Schema

> 注意：关键字段都带有 `description`，让模型仅凭 Schema 就能理解图表形态和填充方式。这是 [3.0 核心设计原则](#30-核心设计原则)的直接体现。

```json
{
  "type": "object",
  "description": "雷达图：在多个维度轴上绘制多边形，对比 1-3 个实体在各维度的能力分布",
  "required": ["type", "label", "radarAxes", "radarEntities"],
  "properties": {
    "type": {
      "type": "string",
      "const": "RadarChart"
    },
    "label": {
      "type": "string",
      "description": "图表标题，描述本次对比的主题"
    },
    "unit": {
      "type": "string",
      "description": "数值单位，如 '%' 或 '分'"
    },
    "radarAxes": {
      "type": "array",
      "description": "雷达图的维度轴名称列表，每个名称对应一个从中心向外的轴",
      "items": {
        "type": "string",
        "maxLength": 10
      },
      "minItems": 3,
      "maxItems": 8,
      "uniqueItems": true
    },
    "radarEntities": {
      "type": "array",
      "description": "参与对比的实体列表，每个实体绘制为一个雷达多边形",
      "items": {
        "$ref": "#/$defs/RadarEntity"
      },
      "minItems": 1,
      "maxItems": 3
    }
  },
  "$defs": {
    "RadarEntity": {
      "type": "object",
      "description": "一个参与对比的实体，其 values 数组在雷达图上形成一个封闭多边形",
      "required": ["label", "values"],
      "properties": {
        "label": {
          "type": "string",
          "description": "实体名称，显示在图例中"
        },
        "values": {
          "type": "array",
          "description": "各维度的数值，顺序与 radarAxes 一一对应，数值越大多边形越向外扩展",
          "items": {
            "type": "number",
            "minimum": 0,
            "maximum": 100
          }
        },
        "color": {
          "type": "string",
          "pattern": "^#[0-9A-Fa-f]{6}$"
        },
        "highlight": {
          "type": "boolean",
          "default": false
        }
      }
    }
  }
}
```

### 2.3 AI 使用说明

**选型一行描述：**

```
| RadarChart | Multi-axis comparison | radarAxes[], radarEntities[{label,values[],highlight?}] |
```

**使用边界：**

| 维度 | 说明 |
|---|---|
| 适用于 | 需要在 3 个以上维度对比 1-3 个实体的能力分布或属性差异 |
| 不适用于 | 仅对比 2 个维度（用 ComparisonBar）；实体超过 3 个（用 BenchmarkHeatmap）；纯排名（用 RankingBar） |

**填充约束：**
- `radarEntities[].values` 长度必须与 `radarAxes` 长度一致
- 所有 `values` 数值应在同一量级范围内（如 0-100）
- `radarAxes` 中的维度名称应简短（2-4 字为佳）

### 2.4 边界情况处理

| 异常场景 | 处理方式 |
|---|---|
| `values` 长度与 `radarAxes` 不匹配 | 截取较短一方的长度，缺失维度显示为 0 |
| `values` 中含 null | 该维度值视为 0 |
| `radarAxes` 少于 3 个 | 降级为柱状图显示 |
| 实体超过 3 个 | 只显示前 3 个 |

## 3. Schema 设计指南

> 本节是 Schema 设计的详细规则参考。如果你已通读了第 2 节的完整示例，可以在实际设计时按需查阅。

### 3.0 核心设计原则

> **模型看到 Schema 就能想象出图表的样子，知道是做什么用的，知道该如何填充数据。**

这是 Schema 设计的第一性原理。所有设计决策——字段命名、结构层级、类型约束——都应服务于这一目标。

| 维度 | 含义 | 检验方法 |
|---|---|---|
| **可视化性** | 字段名和结构让模型推断出图表的视觉形态 | 只看字段名，能否描述出图表长什么样？ |
| **目的性** | Schema 让模型理解图表解决什么分析问题 | 只看 Schema，能否判断何时该用这个图表？ |
| **可填充性** | 类型约束、值域、数组长度让模型知道每个字段该填什么 | 只看 Schema，能否直接填出一份合法数据？ |

**这条原则如何落地到每个设计环节：**

- **字段命名即语义**：`radarAxes` 让模型联想到雷达图的各个轴，`funnelStages` 让模型联想到漏斗的各层级——而不是用 `items` 或 `data` 这类抽象名称
- **结构即形态**：数组套对象的层级关系暗示了图表的视觉层级。例如 `radarEntities[{values[]}]` → 多个重叠的雷达多边形；`nodes[] + links[]` → 网络拓扑
- **约束即指引**：`minItems: 3, maxItems: 8` 告诉模型"3-8 个是合理的"；`minimum: 0, maximum: 100` 暗示"这是百分制评分"
- **description 即说明**：在 JSON Schema 中为关键字段添加 `description`，直接用自然语言告诉模型字段的用途和填充方式（见 [2.2 节示例](#json-schema)和 [3.7 节映射规则](#37-从表格到-json-schema)）

**反面示例对照：**

| 设计方式 | 模型能理解吗？ | 问题 |
|---|---|---|
| `data: [{a: "Q1", b: 12.5}]` | ❌ | 字段名无语义，模型不知道 a 和 b 代表什么 |
| `values: [{label: "Q1", value: 12.5}]` | ✅ | 结构清晰，一看就知道是"标签-数值"对 |
| `items: [{label, score}]`（用于雷达图） | ⚠️ | `items` 太泛化，模型无法区分这是雷达图还是柱状图 |
| `radarEntities: [{label, values[]}]` | ✅ | 前缀 `radar` + `values[]`（多维数值）→ 明确是雷达图的实体 |

### 3.1 从视觉到数据——如何拆解字段

设计 Schema 最大的难点不是格式，而是"我需要哪些字段"。以下三步帮你从图表的视觉效果反推出数据结构：

**第一步：标注视觉元素**

画出（或想象）图表的样子，标注出所有需要"数据驱动"的视觉元素。

以柱状图为例：
- 图表标题 → 需要一个字段
- 每根柱子的标签 → 需要一个字段
- 每根柱子的高度 → 需要一个字段
- Y 轴的单位 → 需要一个字段

**第二步：判断结构——扁平 or 嵌套**

| 视觉特征 | 数据结构 | 示例 |
|---|---|---|
| 只有一个主值 | 扁平字段：`value` | `value: "$47.5B"` |
| 多个同类项 | 数组：`items[]` | `values: [{label, value}, ...]` |
| 每项有多个属性 | 对象数组：`items[{...}]` | `networkNodes: [{id, label, tier, type}, ...]` |
| 有两个平行主体 | 两个平行字段 | `entityA` + `entityB` |
| 有两层维度 | 外层数组 + 内层数组 | `dimensions[]` + `entities[{scores[]}]` |

**第三步：检查"这份数据是否足够渲染图表"**

用你设计的 Schema 手动填一份 JSON 数据，然后问自己：
- 光看这份 JSON，能不能把图表画出来？
- 每个视觉元素都有对应的数据来源吗？
- 有没有不需要的冗余字段？

### 3.2 公共字段

所有图表都包含以下公共字段，不需要额外设计：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `type` | `string` | 是 | 图表类型名，与外层 type 一致 |
| `label` | `string` | 是 | 图表标题 |
| `unit` | `string` | 否 | 数值单位（见下方 unit 约定） |

**unit 字段约定**

| 格式 | 含义 | 显示效果 |
|---|---|---|
| `"$B"` | 十亿美元 | `$12.5B` |
| `"$M"` | 百万美元 | `$350M` |
| `"%"` | 百分比 | `42%` |
| `"x"` | 倍数 | `4.2x` |
| `"K"` | 千 | `120K` |

### 3.3 字段类型速查

| 类型 | 通俗含义 | 示例 |
|---|---|---|
| `string` | 一段文本 | `"营收增长"` |
| `number` | 一个数字 | `42.5` |
| `boolean` | 是或否 | `true`（是）/ `false`（否） |
| `string[]` | 一组文本（列表） | `["推理", "编码", "创意"]` |
| `{label, value}` | 一组搭配信息（对象） | `{"label": "Q1", "value": 12.5}` |
| `{label, value}[]` | 多组搭配信息（对象列表） | 多个上述对象 |

> **符号提示：** `[]` 表示"一组/多个"，`{}` 表示"一条包含多个属性的记录"，`?` 表示"可选"。

### 3.4 字段命名规则与前缀

| 规则 | 正确示例 | 错误示例 |
|---|---|---|
| camelCase 命名 | `radarAxes` | `radar_axes`, `RadarAxes` |
| 带图表前缀避免冲突 | `radarEntities` | `entities`（与其他图表冲突） |
| 数组字段名用复数 | `radarAxes` | `radarAxis` |
| 布尔字段用形容词 | `highlight` | `isHighlight` |
| 可选标签/描述字段 | `sublabel`, `subtitle` | `desc`（太模糊） |

**为什么要带图表前缀？** 所有图表类型共用一个 `ChartData` 数据结构。如果不同图表使用同名字段但含义不同，就会产生冲突。

**判断原则：**
- 字段结构（子字段含义或个数）与已有字段不同 → **必须加图表前缀**
- 数据结构与现有字段完全一致（如 `values[{label, value}]`）→ 可以直接复用，无需新建

> **提示：** 查阅 [chart-examples.md](./chart-examples.md) 中的图表总览表，了解已有字段名以避免冲突。

### 3.5 自定义子类型

当数据项有多个属性时，需要定义一个子类型。用表格描述即可（完整示例见 [2.2 节 RadarEntity](#子类型radrarentity)）：

| 字段 | 类型 | 必填 | 说明 | 约束 |
|---|---|---|---|---|
| `label` | `string` | 是 | 项目名称 | — |
| `values` | `number[]` | 是 | 数值列表 | 根据业务含义设定范围 |
| `color` | `string` | 否 | 自定义颜色 | HEX 格式 |
| `highlight` | `boolean` | 否 | 是否高亮 | 默认 false |

### 3.6 数据量建议

每个图表 Schema 都应标注建议的数据量范围：

- **最少数据点** — 低于此数量图表没有意义
- **建议范围** — 最佳展示效果
- **上限** — 超过此数量会导致视觉拥挤

### 3.7 从表格到 JSON Schema

你用表格设计的 Schema 最终会被转化为正式的 JSON Schema。了解这个格式有助于你更精确地表达设计意图。

**表格 → JSON Schema 映射规则**

| 表格中的描述 | JSON Schema 对应 |
|---|---|
| 字段名 `radarAxes` | `"radarAxes": { ... }` |
| 类型 `string` | `"type": "string"` |
| 类型 `number` | `"type": "number"` |
| 类型 `boolean` | `"type": "boolean"` |
| 类型 `string[]` | `"type": "array", "items": { "type": "string" }` |
| 类型 `{label, value}[]` | `"type": "array", "items": { "type": "object", "properties": { ... } }` |
| 必填 = 是 | 字段名出现在 `"required": [...]` 中 |
| 约束：3-8 个 | `"minItems": 3, "maxItems": 8` |
| 约束：0-100 | `"minimum": 0, "maximum": 100` |
| 约束：不超过 10 字 | `"maxLength": 10` |
| 约束：不允许重复 | `"uniqueItems": true` |
| 默认值 | `"default": false` |
| 子类型引用 | `"$ref": "#/$defs/TypeName"` |
| HEX 颜色格式 | `"pattern": "^#[0-9A-Fa-f]{6}$"` |
| 字段说明文本 | `"description": "..."` |
| 顶层对象说明 | 根对象 `"description": "图表类型：一句话描述视觉形态和用途"` |

**逐字段对照示例**

以 `radarAxes` 字段为例，看表格描述如何映射到 JSON Schema：

| 表格列 | 表格中的值 | JSON Schema 中的表达 |
|---|---|---|
| 字段名 | `radarAxes` | 作为 `properties` 的 key |
| 类型 | `string[]` | `"type": "array"` + `"items": {"type": "string"}` |
| 必填 | 是 | 出现在 `"required"` 数组中 |
| 说明 | 维度名称列表 | `"description": "雷达图的维度轴名称列表，每个名称对应一个从中心向外的轴"` |
| 约束：3-8 个 | 3-8 个 | `"minItems": 3, "maxItems": 8` |
| 约束：不允许重复 | 不允许重复 | `"uniqueItems": true` |
| 约束：不超过 10 字 | 不超过 10 字 | `"items"` 中 `"maxLength": 10` |

完整的 JSON Schema 示例见 [2.2 节 RadarChart](#json-schema)。

> **给 PM 的提示：** 你不需要自己写 JSON Schema——这是工程师的工作。但理解这个格式可以帮你在表格中更精确地表达约束，减少与工程师之间的沟通成本。
>
> **关于 `description` 字段：** 请在表格的"说明"列中写清楚每个字段的用途和视觉含义，工程师会将这些说明转化为 JSON Schema 的 `description`。好的 description 应该让模型读完就能"脑补"出图表的样子——这是 [3.0 核心设计原则](#30-核心设计原则)最直接的落地方式。

## 4. AI 使用说明设计指南

AI 使用说明决定了 AI 能否正确选用和填充你的图表。不要低估这部分的重要性——即使 Schema 设计完美，如果 AI 不理解何时使用、如何填充，图表也无法正确生成。

### 4.1 需要提供的内容

为每个新图表编写以下结构化说明：

**① 图表选型一行描述**（给 chart-middleware 的注册信息）

```
| 图表类型名 | 适用场景（英文简述） | 关键字段简写 |
```

**② 使用边界说明**（什么时候该用 / 不该用）

| 维度 | 说明 |
|---|---|
| **适用于** | 描述适合使用此图表的条件 |
| **不适用于** | 描述不适合的场景，并指出应该用哪个替代图表 |

**③ 数据填充约束**（AI 填写数据时必须遵守的规则）

例如：
- 数组之间的长度对齐要求
- 数值的量级范围
- 文本的长度建议

完整示例见 [2.3 节](#23-ai-使用说明)。

### 4.2 常见 AI 误用模式

设计时预想以下误用场景，并在说明中明确提醒：

| 误用模式 | 说明 | 预防方式 |
|---|---|---|
| **类型混淆** | AI 选了一个相似但不合适的图表类型 | 在"不适用于"中明确与相似图表的边界 |
| **数组长度不匹配** | 多个关联数组的长度不一致 | 在约束中明确"长度必须等于 X" |
| **超出数据量限制** | AI 生成了过多的数据项 | 在约束中标注上限并说明理由 |
| **缺少必填字段** | AI 遗漏了必填字段 | 在关键字段简写中不加 `?` 标记 |
| **值域不合理** | 数值超出合理范围 | 明确标注值域范围（如 0-100） |

## 5. 边界情况与异常数据处理

设计 Schema 时，必须考虑 AI 生成数据可能出现的各种异常。对于每个字段，逐项排查以下场景：

### 5.1 异常数据清单

| 异常场景 | 你需要决定 | 示例 |
|---|---|---|
| **必填字段缺失** | 前端如何降级？是否显示占位符？ | `label` 为空时，显示 "Untitled" |
| **数值为 null/undefined** | 该数据点是隐藏还是显示为 0？ | 柱状图中某个 value 为 null，该柱子不显示 |
| **数值为 0** | 0 是有意义的数据还是无效数据？ | 漏斗图中某阶段值为 0，仍显示该阶段 |
| **数值为负数** | 图表是否支持负数？如何展示？ | 瀑布图支持负数（红色柱子），饼图不支持 |
| **文本过长** | 超长标签如何截断？ | 标签超过 15 字时截断并加 `...` |
| **数组为空** | 空数组是否合法？显示什么？ | `values: []` 时显示"暂无数据"提示 |
| **数据量超限** | 超过推荐上限如何处理？ | 柱状图超过 8 项时，只显示前 8 项 |
| **重复值** | 是否允许相同的 label？ | 允许，但可能导致图例混淆 |

### 5.2 如何在 Schema 中体现

对于每个需要处理的边界情况，在 Schema 定义表格的"约束"列中说明即可。完整示例见 [2.4 节](#24-边界情况处理)。

### 5.3 默认值约定

对于可选字段，必须明确默认值：

| 字段 | 默认值 | 说明 |
|---|---|---|
| `highlight` | `false` | 不高亮 |
| `color` | 系统配色方案自动分配 | 不指定时按顺序使用默认色板 |
| `unit` | 无 | 不显示单位 |
| `comparison` | 无 | 不显示对比信息 |

## 6. 设计自检 Checklist

完成图表设计后，逐项确认：

### 核心原则验证（最重要）

- [ ] **可视化性**：只看字段名和结构，能否描述出图表长什么样？
- [ ] **目的性**：只看 Schema，能否判断这个图表适用于什么分析场景？
- [ ] **可填充性**：只看 Schema（含 description 和约束），能否直接填出一份合法数据？
- [ ] 顶层对象有 `description`，一句话说明图表的视觉形态和用途
- [ ] 关键字段都有 `description`，说明字段的视觉含义和填充方式

### Schema 完整性

- [ ] 所有视觉元素都有对应的数据字段
- [ ] 每个字段都标注了类型、是否必填、约束
- [ ] 可选字段都有明确的默认值
- [ ] 字段名使用 camelCase 格式
- [ ] 与已有图表有冲突的字段名加了图表前缀
- [ ] 提供了完整的 JSON Schema

### 数据模型合理性

- [ ] 用设计的 Schema 手写了一份 JSON 数据，能完整渲染图表
- [ ] 数据量建议（最小/推荐/最大）已标注
- [ ] 没有冗余字段——每个字段都对应一个视觉元素

### AI 使用说明

- [ ] 提供了选型一行描述
- [ ] 明确了"适用于"和"不适用于"的场景边界
- [ ] 标注了数据填充约束（数组长度、值域范围等）
- [ ] 预想了常见 AI 误用场景并在约束中预防

### 边界情况

- [ ] 考虑了必填字段缺失时的降级方案
- [ ] 考虑了空数组、null 值、0 值的处理
- [ ] 考虑了文本过长的截断策略
- [ ] 考虑了数据量超限的处理

### 测试数据

- [ ] 提供了至少 1 个典型场景的完整 JSON
- [ ] 提供了至少 1 个边界场景（最小数据量或含可选字段）
- [ ] 测试数据符合 Schema 定义的所有约束

### 与现有图表的关系

- [ ] 确认没有与现有图表功能完全重复
- [ ] 说明了与相似图表的区别和各自适用场景

## 附录 B：颜色与视觉约定

### 默认配色方案

系统提供默认色板，PM 通常**不需要**在 Schema 中指定颜色。以下情况除外：

| 场景 | 是否需要 PM 指定颜色 | 说明 |
|---|---|---|
| 普通数据系列 | 否 | 系统自动按顺序分配默认色板 |
| 语义色彩（涨/跌、正/负） | 否 | 系统自动处理（绿色=正向，红色=负向） |
| 品牌色要求 | 可选 | 如特定实体需要品牌色，可加 `color` 字段 |
| 高亮标记 | 可选 | 通过 `highlight: true` 实现，无需指定颜色 |

### color 字段规范

如果你的图表确实需要自定义颜色字段：

- 使用 HEX 格式：`"#FF0000"`（6 位，不含透明度）
- 设为**可选字段**，提供系统默认色作为降级
- 限制在子类型中使用（如 `RadarEntity.color`），不放在顶层

### highlight 字段规范

`highlight` 是一个通用的视觉强调机制：

- 类型：`boolean`，可选，默认 `false`
- 效果：前端会以更饱和的颜色、更粗的线条或边框来突出该项
- 建议每个图表中高亮项不超过 1-2 个

## 附录 C：术语表

| 术语 | 含义 |
|---|---|
| **JSON Schema** | 对 JSON 数据的"规格说明"——定义有哪些字段、每个字段的类型、约束和是否必填。类似空白表单的模板设计 |
| **PascalCase** | 每个单词首字母大写、无分隔符的命名格式。例：`RadarChart`、`NetworkGraph` |
| **camelCase** | 第一个单词小写，后续单词首字母大写的命名格式。例：`radarAxes`、`networkNodes` |
| **HEX** | 十六进制颜色代码格式，以 `#` 开头。例：`#FF0000`（红色）、`#00FF00`（绿色） |
| **子类型** | 当一个字段的值是包含多个属性的对象时，为该对象定义的结构说明 |
| **降级（Fallback）** | 当数据异常时，系统退而求其次的展示方式 |
| **前缀（Prefix）** | 在字段名前添加图表类型标识，避免不同图表间的字段名冲突 |
| **`required`** | JSON Schema 中标记必填字段的关键字 |
| **`$ref` / `$defs`** | JSON Schema 中引用子类型定义的语法，用于复用结构 |
