# Edgen 业务组件设计指南

> 面向产品经理：如何设计和定义业务组件（复杂组件 & 简单引导组件）

---

## 概要

本指南帮助产品经理设计**业务组件**——与通用图表不同，业务组件不需要 AI 生成完整数据，而是由 AI 传入少量参数，前端负责后续处理。

业务组件分为两个子类：

| 子类 | 特征 | 典型场景 |
|---|---|---|
| **复杂组件** | 复用产品已有组件，或新设计带复杂交互的卡片；AI 传参，前端调 API 取实时数据后渲染 | 历史价格走势图、板块热力图 |
| **简单引导组件** | 单一行为——点击即跳转；不同跳转类型可以有不同卡片样式 | 跳转到个股详情、跳转到板块页、跳转到功能页 |

| 交付物 | 说明 | 对应章节 |
|---|---|---|
| **需求澄清** | 组件名称、用途、与通用图表的区别 | [1.5](#15-设计流程概览) |
| **Schema 定义** | 参数表格 + JSON Schema | [3](#3-schema-设计指南复杂组件) / [7](#7-简单引导组件设计指南) |
| **AI 触发说明** | 意图识别规则、参数提取指引 | [4](#4-ai-触发说明设计指南) |
| **边界情况处理** | 参数缺失、API 异常的降级方案 | [5](#5-边界情况与异常数据处理) |
| **测试数据** | 1-2 个完整 JSON 示例 | [1.5](#15-设计流程概览) |

如果你是第一次设计业务组件，建议先看[第 2 节的完整示例](#2-完整示例复杂组件)了解最终交付物的全貌。

如果你需要设计的是通用图表（AI 直接填充所有数据），请参考 [chart-design-guide.md](./chart-design-guide.md)。

## 1. 什么是业务组件

### 1.1 业务组件 vs 通用图表

| | 通用图表 | 业务组件 |
|---|---|---|
| **AI 的角色** | 数据生产者——填充全部图表数据 | 意图指定者——传入查询参数 |
| **Schema 内容** | 完整的数据结构（labels, values, axes...） | 查询参数（assetId, dateRange, market...） |
| **数据来源** | AI 的知识与推理 | 前端调用后端 API，获取实时数据 |
| **数据时效性** | 受限于 AI 训练数据的截止时间 | 实时或准实时 |
| **典型场景** | "对比 A 和 B 的收入"→ AI 查知识库构造数据 | "看一下苹果最近的股价"→ AI 识别资产 + 时间范围，前端拉取行情 |

**判断标准：选通用图表还是业务组件？**

| 问题 | 选通用图表 | 选业务组件 |
|---|---|---|
| 数据需要实时吗？ | 不需要，历史数据或 AI 推理即可 | 需要，必须是最新数据 |
| AI 能生成这些数据吗？ | 能，AI 有足够的知识 | 不能或不应该，数据量大/需要精确/变动频繁 |
| 数据结构固定吗？ | 不固定，每次内容不同 | 固定，同类组件结构一致 |

### 1.2 复杂组件 vs 简单引导组件

业务组件内部分为两个子类，设计方式和复杂度有明显差异：

| | 复杂组件 | 简单引导组件 |
|---|---|---|
| **定义** | 复用产品已有组件，或新设计带复杂交互的卡片 | 单一行为——点击即跳转到目标页面 |
| **交互复杂度** | 高：可切换时间范围、切换指标、悬停查看详情等 | 低：仅点击跳转，无其他交互 |
| **前端行为** | 调用 API 获取实时数据 → 渲染为完整功能组件 | 渲染为一张样式卡片 → 点击后导航到目标页 |
| **Schema 设计重心** | 查询参数——告诉前端"去哪取什么数据" | 跳转目标——告诉前端"点击后跳到哪里" |
| **卡片样式** | 由组件自身决定，各不相同 | 由跳转类型决定，同类跳转样式统一 |
| **AI 的参数量** | 中等（2-5 个查询参数） | 极少（1-3 个：跳转类型 + 目标 ID + 可选展示信息） |
| **典型场景** | "看一下苹果的股价走势" → PriceHistoryChart | "帮我打开苹果的详情页" → 跳转引导卡片 |

**判断标准：选复杂组件还是简单引导组件？**

| 问题 | 选复杂组件 | 选简单引导组件 |
|---|---|---|
| 需要在组件内展示数据或图表吗？ | 是，组件本身就是一个数据视图 | 否，只需引导用户跳转到目标页 |
| 用户需要在组件内进行交互吗？ | 是，如切换时间范围、切换指标 | 否，唯一操作就是点击跳转 |
| 组件需要丰富的交互能力吗？ | 是，需要切换、筛选、悬停等多种交互 | 否，只需要一张引导卡片 |

### 1.3 工作原理

**复杂组件：**

```
用户提问 → AI 识别意图 → AI 输出组件参数 JSON → 前端解析参数 → 前端调 API 取数据 → 渲染组件
```

**简单引导组件：**

```
用户提问 → AI 识别意图 → AI 输出跳转参数 JSON → 前端渲染引导卡片 → 用户点击 → 跳转到目标页
```

两者的共同点：AI 不生产数据，只输出"指令"。区别在于复杂组件的指令是"去哪取什么数据"，简单引导组件的指令是"跳到哪个页面"。

### 1.4 你需要做什么 / 不需要做什么

| 你的职责 | 不需要关心 |
|---|---|
| 明确组件用途——解决什么场景 | 前端组件的实现代码 |
| 设计参数 Schema——定义 AI 需要传什么参数 | API 接口的具体实现 |
| 编写 AI 触发说明——什么对话意图应触发此组件 | 数据加载与缓存策略 |
| 考虑边界情况——参数缺失、API 失败时的处理 | 组件的动画与交互细节 |

### 1.5 组件数据的整体结构

业务组件与通用图表共享同一个外层格式，区别仅在 `chartData` 内部：

```json
{
  "root": "chart",
  "elements": {
    "chart": {
      "type": "你的组件类型名",
      "props": {
        "chartData": {
          "type": "你的组件类型名",
          "label": "组件标题",
          "...": "← 你设计的查询参数放这里"
        }
      },
      "children": []
    }
  }
}
```

- `type`：组件类型名称，PascalCase 格式（如 `PriceHistoryChart`）
- `label`：组件标题，必填——AI 根据用户意图生成

## 2. 完整示例：复杂组件

### 2.1 PriceHistoryChart（历史价格）

#### 需求澄清

| 问题 | 回答 |
|---|---|
| 组件名称 | `PriceHistoryChart`（历史价格走势图） |
| 一句话描述 | 展示某个资产在指定时间范围内的历史价格走势 |
| 典型场景 | "看一下苹果最近的股价"、"比特币过去一年的走势"、"特斯拉上市以来的价格" |
| 为什么不用通用 LineChart | 行情数据量大（数百个数据点）、需要实时、AI 无法精确提供每日价格 |
| 前端需要调用的 API | 行情数据服务（传入 assetId + dateRange，返回价格时间序列） |

#### 参数 Schema

| 字段 | 类型 | 必填 | 说明 | 约束 |
|---|---|---|---|---|
| `assetId` | `string` | 是 | 资产唯一标识符 | 如股票代码 "AAPL"、加密货币 "BTC" |
| `assetName` | `string` | 否 | 资产显示名称 | 用于组件标题，如 "Apple Inc." |
| `dateRange` | `string` | 否 | 时间范围 | 枚举值，默认 `"1Y"` |

**dateRange 枚举值：**

| 值 | 含义 |
|---|---|
| `"1D"` | 最近 1 天 |
| `"1W"` | 最近 1 周 |
| `"1M"` | 最近 1 个月 |
| `"3M"` | 最近 3 个月 |
| `"6M"` | 最近 6 个月 |
| `"1Y"` | 最近 1 年（默认） |
| `"5Y"` | 最近 5 年 |
| `"ALL"` | 上市以来全部 |

#### JSON Schema

```json
{
  "type": "object",
  "description": "历史价格走势：前端根据 assetId 和 dateRange 调用行情 API，渲染为带时间轴的价格折线图",
  "required": ["type", "label", "assetId"],
  "properties": {
    "type": {
      "type": "string",
      "const": "PriceHistoryChart"
    },
    "label": {
      "type": "string",
      "description": "组件标题，由 AI 根据上下文生成，如 'Apple 近一年股价走势'"
    },
    "assetId": {
      "type": "string",
      "description": "资产唯一标识符，如股票代码 'AAPL' 或加密货币代码 'BTC'"
    },
    "assetName": {
      "type": "string",
      "description": "资产显示名称，如 'Apple Inc.'，用于组件内的标题和图例"
    },
    "dateRange": {
      "type": "string",
      "description": "价格数据的时间范围，决定 X 轴跨度和数据粒度",
      "enum": ["1D", "1W", "1M", "3M", "6M", "1Y", "5Y", "ALL"],
      "default": "1Y"
    }
  }
}
```

#### AI 触发说明

**触发意图：**

用户想查看某个资产的历史价格走势。常见表述：

- "看一下 XX 的股价"
- "XX 最近走势怎么样"
- "XX 过去 N 年的价格"
- "XX 的 K 线 / 行情 / 走势图"

**参数提取规则：**

| 参数 | 提取方式 | 提取不到时 |
|---|---|---|
| `assetId` | 从用户提到的资产名称匹配对应代码 | 必须提取到，否则追问用户 |
| `assetName` | 从用户提到的资产名称或 AI 的知识中获取 | 可省略 |
| `dateRange` | 从"最近一年"、"过去三个月"等时间表述推断 | 默认 `"1Y"` |

**不适用于：**

- 用户想对比两个资产的价格 → 应分别生成两个 PriceHistoryChart
- 用户想看基本面数据（市值、PE 等）→ 用通用图表 StatGrid
- 用户想看技术指标分析 → 暂不支持

#### 边界情况处理

| 异常场景 | 处理方式 |
|---|---|
| `assetId` 无法识别 | AI 应追问用户确认资产名称，不输出组件 |
| `assetId` 合法但 API 返回无数据 | 前端显示"暂无该资产的行情数据" |
| `dateRange` 超出资产上市时间 | 前端自动截取到可用范围 |
| API 超时或失败 | 前端显示重试按钮 |

```json
        }
      },

    }
  }

```

```json

      "props": {

          "label": "比特币近三个月走势",
          "assetId": "BTC",
          "assetName": "Bitcoin",
          "dateRange": "3M"
        }
      },
      "children": []
    }
  }
}
```

### 2.2 SectorHeatmap（板块热力图）

#### 需求澄清

| 问题 | 回答 |
|---|---|
| 组件名称 | `SectorHeatmap`（板块热力图） |
| 一句话描述 | 以色块矩阵展示某个市场各板块的实时涨跌情况，红涨绿跌（或绿涨红跌），面积代表市值权重 |
| 典型场景 | "今天美股各板块表现怎么样"、"看一下加密货币市场热力图"、"港股哪些板块在涨" |
| 为什么不用通用 TreemapChart | 板块数量多（几十个）、数据需实时、每个色块需包含涨跌幅/市值等多维信息 |
| 前端需要调用的 API | 板块行情服务（传入 market，返回各板块名称、涨跌幅、市值权重） |

#### 参数 Schema

| 字段 | 类型 | 必填 | 说明 | 约束 |
|---|---|---|---|---|
| `market` | `string` | 是 | 市场类型 | 枚举值 |
| `metric` | `string` | 否 | 色块着色依据 | 枚举值，默认 `"dailyChange"` |

**market 枚举值：**

| 值 | 含义 |
|---|---|
| `"us_stock"` | 美股市场 |
| `"hk_stock"` | 港股市场 |
| `"crypto"` | 加密货币市场 |

**metric 枚举值：**

| 值 | 含义 |
|---|---|
| `"dailyChange"` | 当日涨跌幅（默认） |
| `"weeklyChange"` | 本周涨跌幅 |
| `"monthlyChange"` | 本月涨跌幅 |

#### JSON Schema

```json
{
  "type": "object",
  "description": "板块热力图：前端根据 market 调用板块行情 API，以矩形色块矩阵展示各板块涨跌，面积表示市值权重，颜色深浅表示涨跌幅度",
  "required": ["type", "label", "market"],
  "properties": {
    "type": {
      "type": "string",
      "const": "SectorHeatmap"
    },
    "label": {
      "type": "string",
      "description": "组件标题，由 AI 根据上下文生成，如 '美股板块今日表现'"
    },
    "market": {
      "type": "string",
      "description": "目标市场，决定从哪个市场的 API 拉取板块数据",
      "enum": ["us_stock", "hk_stock", "crypto"]
    },
    "metric": {
      "type": "string",
      "description": "色块颜色的度量依据，涨为暖色、跌为冷色",
      "enum": ["dailyChange", "weeklyChange", "monthlyChange"],
      "default": "dailyChange"
    }
  }
}
```

#### AI 触发说明

**触发意图：**

用户想了解某个市场各板块/行业的整体涨跌情况。常见表述：

- "今天美股各板块表现怎么样"
- "看一下加密货币市场热力图"
- "港股哪些板块在涨"
- "市场概览"、"大盘全景"

**参数提取规则：**

| 参数 | 提取方式 | 提取不到时 |
|---|---|---|
| `market` | 从"美股"、"港股"、"加密货币/币圈"等关键词匹配 | 根据对话上下文推断；仍无法确定则追问用户 |
| `metric` | 从"今天"→ dailyChange、"这周"→ weeklyChange、"这个月"→ monthlyChange 推断 | 默认 `"dailyChange"` |

**不适用于：**

- 用户想看单个股票的详情 → 用 PriceHistoryChart
- 用户想对比两个板块的历史走势 → 用通用 LineChart
- 用户想看某板块内的个股排名 → 暂不支持（可后续扩展）

#### 边界情况处理

| 异常场景 | 处理方式 |
|---|---|
| `market` 不在枚举范围内 | AI 应映射到最接近的枚举值，或追问用户 |
| API 返回空数据（如非交易时段） | 前端显示"当前无交易数据，展示最近交易日数据" |
| API 超时或失败 | 前端显示重试按钮 |

## 3. Schema 设计指南（复杂组件）

### 3.0 核心设计原则

与通用图表相同的第一性原理：

> **模型看到 Schema 就能想象出组件呈现什么信息，知道什么对话意图应该触发它，知道该如何从用户话语中提取参数。**

| 维度 | 含义 | 检验方法 |
|---|---|---|
| **可视化性** | 模型能推断出组件呈现什么信息 | 只看 Schema，能否描述出组件展示什么内容？ |
| **目的性** | 模型知道什么对话意图应触发此组件 | 只看 Schema，能否判断用户说什么话时该用它？ |
| **可填充性** | 模型知道如何从对话中提取每个参数 | 只看 Schema，能否从一句用户话中填出合法参数？ |

### 3.1 业务组件 Schema 的特殊要求

业务组件的 Schema 与通用图表有本质区别——**参数少而精，每个参数都是 API 查询的关键输入**。

| 设计原则 | 说明 | 示例 |
|---|---|---|
| **参数最小化** | 只包含前端 API 调用所需的最少参数 | PriceHistoryChart 只需 assetId + dateRange，不需要价格数据 |
| **枚举优先** | 有限选项的参数必须用 `enum` 约束 | `dateRange: enum["1D","1W","1M",...]` 而不是自由文本 |
| **智能默认值** | 每个可选参数都有合理的默认值 | `dateRange` 默认 `"1Y"`，用户不指定时间时自动使用 |
| **语义化命名** | 参数名要让模型一看就知道从对话中提取什么 | `assetId` 而不是 `id`，`market` 而不是 `source` |

### 3.2 公共字段

所有业务组件都包含以下公共字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `type` | `string` | 是 | 组件类型名，PascalCase 格式 |
| `label` | `string` | 是 | 组件标题，AI 根据上下文动态生成 |

注意：业务组件通常**不需要** `unit` 字段（单位由前端从 API 数据中获取）。

### 3.3 参数类型选择

| 参数特征 | 推荐类型 | 示例 |
|---|---|---|
| 有限的固定选项 | `string` + `enum` | `market: enum["us_stock", "hk_stock", "crypto"]` |
| 资产/实体标识 | `string` | `assetId: "AAPL"` |
| 开关类配置 | `boolean` | `showVolume: true` |
| 数量范围 | `integer` + `minimum/maximum` | `topN: 10`（展示前 N 名） |

> **关键区别：** 通用图表的字段通常是 `array`（数据列表），业务组件的字段通常是 `string` 或 `enum`（查询参数）。

### 3.4 字段命名规则

| 规则 | 正确示例 | 错误示例 |
|---|---|---|
| camelCase 命名 | `dateRange` | `date_range`, `DateRange` |
| 参数名暗示"从哪提取" | `assetId`（从用户提到的资产名提取） | `id`（太模糊） |
| 枚举值用 snake_case | `"us_stock"`, `"hk_stock"` | `"USStock"`, `"HK Stock"` |
| 布尔参数用动词或形容词 | `showVolume`, `includePreMarket` | `volume`, `preMarket` |

### 3.5 从表格到 JSON Schema

映射规则与通用图表基本一致（参考 [chart-design-guide.md 3.7 节](./chart-design-guide.md#37-从表格到-json-schema)），额外规则：

| 表格中的描述 | JSON Schema 对应 |
|---|---|
| 枚举值列表 | `"enum": ["value1", "value2", ...]` |
| 默认值 | `"default": "value"` |
| 顶层对象说明 | `"description": "组件名：前端根据 XX 参数调用 XX API，渲染为 XX"` |

**description 的写法要点：**

业务组件的 `description` 需要额外说明**前端做什么**，因为模型需要理解这不是一个它需要填数据的图表：

| 写法 | 好 / 坏 | 原因 |
|---|---|---|
| `"前端根据 assetId 调用行情 API，渲染为价格折线图"` | 好 | 模型知道自己只需要填参数 |
| `"展示资产的历史价格走势"` | 不够好 | 模型可能以为要自己填价格数据 |

## 4. AI 触发说明设计指南

### 4.1 与通用图表的区别

通用图表的 AI 说明关注"选型 + 填充数据"，业务组件的 AI 说明关注"识别意图 + 提取参数"。

| 环节 | 通用图表 | 业务组件 |
|---|---|---|
| **选择时机** | AI 判断"用什么图表展示这些数据" | AI 判断"用户是不是想看这类实时信息" |
| **数据来源** | AI 从知识中构造数据 | AI 从用户对话中提取参数 |
| **输出内容** | 完整的数据 JSON | 少量参数 JSON |

### 4.2 需要提供的内容

为每个业务组件编写以下结构化说明：

**① 触发意图**

列出用户的典型表述模式。这些不是精确的关键词匹配，而是帮助 AI 理解什么类型的对话意图应该触发此组件。

**② 参数提取规则**

对每个参数，说明：

| 列 | 说明 |
|---|---|
| **参数** | 参数名 |
| **提取方式** | AI 从对话中提取此参数的方法 |
| **提取不到时** | 默认值策略，或是否需要追问用户 |

**③ 不适用于**

明确哪些相似但不同的场景不应触发此组件。

完整示例见 [2.1 节](#ai-触发说明) 和 [2.2 节](#ai-触发说明-1)。

### 4.3 参数提取的优先级

当 AI 从对话中提取参数时，应遵循以下优先级：

1. **用户明确指定** → 直接使用（"看苹果过去三个月的股价" → assetId="AAPL", dateRange="3M"）
2. **对话上下文推断** → 从前文推断（之前在聊美股 → market="us_stock"）
3. **使用默认值** → Schema 中定义的 default
4. **追问用户** → 必填参数且无法推断时

### 4.4 常见 AI 误用模式

| 误用模式 | 说明 | 预防方式 |
|---|---|---|
| **混淆通用图表和业务组件** | AI 用 LineChart 手动填股价数据 | 在触发说明中明确"涉及实时行情时必须用业务组件" |
| **参数格式错误** | AI 传 `"Apple"` 而不是 `"AAPL"` 作为 assetId | 在提取规则中说明"需要映射为标准代码" |
| **该追问时没追问** | 用户说"看一下股价"但没说哪只股票 | 在提取规则中标注"必填参数提取不到时必须追问" |
| **不该追问时追问** | 用户没指定时间范围，AI 追问而不是用默认值 | 在提取规则中标注可选参数的默认值 |

## 5. 边界情况与异常数据处理

业务组件的边界情况与通用图表不同，主要集中在**参数异常**和 **API 异常**两类。

### 5.1 参数异常

| 异常场景 | 你需要决定 | 示例 |
|---|---|---|
| **必填参数缺失** | AI 应追问还是使用降级方案？ | `assetId` 缺失 → AI 必须追问 |
| **参数值不在枚举范围** | 前端如何处理？ | `market: "a_stock"` 不在枚举中 → 前端忽略并显示提示 |
| **参数值合法但无意义** | 是否需要提示？ | `dateRange: "1D"` 但资产是日更新频率 → 前端照常展示 |

### 5.2 API 异常

| 异常场景 | 你需要决定 | 示例 |
|---|---|---|
| **API 返回空数据** | 显示什么？ | "暂无该资产的行情数据" |
| **API 超时** | 是否自动重试？显示什么？ | 显示重试按钮 |
| **API 返回错误** | 是否显示错误详情？ | 显示通用错误提示，不暴露技术细节 |
| **非交易时段请求** | 显示最新可用数据还是提示？ | 显示最近交易日数据 + 时间标注 |

### 5.3 加载状态

业务组件与通用图表的另一个关键区别：业务组件有**加载等待**阶段。需要为每个组件设计：

| 状态 | 说明 |
|---|---|
| **加载中** | 显示骨架屏或 loading 动画 |
| **加载成功** | 正常渲染组件 |
| **加载失败** | 显示错误提示 + 重试按钮 |
| **无数据** | 显示空状态提示 |

## 7. 简单引导组件设计指南

### 7.1 什么是简单引导组件

简单引导组件的核心行为只有一个：**用户点击 → 跳转到目标页面**。

它在对话流中表现为一张样式卡片，AI 只需要告诉前端"这张卡片点击后跳到哪里"。不同的跳转类型可以有不同的卡片样式（如个股卡片、板块卡片、功能入口卡片），但交互行为统一为点击跳转。

**与复杂组件的关键区别：**

| 维度 | 复杂组件 | 简单引导组件 |
|---|---|---|
| 前端是否需要调 API 取数据 | 是 | 否（或仅取少量展示信息） |
| 组件内是否有交互 | 是（切换、悬停等） | 否（仅点击跳转） |
| 渲染复杂度 | 高 | 低——一张卡片 |
| Schema 设计重心 | 查询参数 | 跳转目标描述 |

### 7.2 核心设计原则

> **模型看到 Schema 就能知道：这是一个引导用户跳转的卡片，知道什么对话意图触发它，知道跳转到哪里。**

| 维度 | 含义 | 检验方法 |
|---|---|---|
| **跳转目标明确** | Schema 清楚描述了点击后跳到哪个页面 | 只看 JSON，能否说出用户点击后会看到什么？ |
| **目的性** | 模型知道什么对话意图触发此引导卡片 | 只看 Schema，能否判断用户说什么话时该用它？ |
| **样式可区分** | 不同跳转类型的卡片在视觉上有区分 | 只看 `type`，能否想象出卡片的样式风格？ |

### 7.3 通用 Schema 结构

所有简单引导组件共享以下 Schema 骨架：

```json
{
  "root": "chart",
  "elements": {
    "chart": {
      "type": "引导组件类型名",
      "props": {
        "chartData": {
          "type": "引导组件类型名",
          "label": "AI 生成的引导文案",
          "action": "navigate",
          "target": {
            "type": "跳转目标类型",
            "id": "目标 ID（可选）",
            "params": {}
          }
        }
      },
      "children": []
    }
  }
}
```

#### chartData 公共字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `type` | `string` | 是 | 引导组件类型名，PascalCase 格式 |
| `label` | `string` | 是 | 卡片标题/引导文案，由 AI 根据上下文生成 |
| `action` | `string` | 是 | 行为类型，固定为 `"navigate"` |
| `target` | `object` | 是 | 跳转目标描述 |

#### target 子结构

| 字段 | 类型 | 必填 | 说明 | 约束 |
|---|---|---|---|---|
| `type` | `string` | 是 | 跳转目标类型 | 枚举值，由各具体引导组件定义 |
| `id` | `string` | 视类型而定 | 目标页面的唯一标识 | 如资产代码、板块 ID、文章 ID 等 |
| `params` | `object` | 否 | 附加参数 | 如默认标签页、锚点位置等 |

#### 卡片展示信息（可选）

为了让卡片在渲染时有更好的展示效果，可以添加以下可选字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `displayTitle` | `string` | 否 | 卡片上显示的标题（与 label 不同，label 是引导文案，displayTitle 是目标页面的名称） |
| `displaySubtitle` | `string` | 否 | 卡片上显示的副标题/补充说明 |
| `displayIcon` | `string` | 否 | 卡片图标类型（由前端映射到具体图标） |

### 7.4 设计新引导组件的步骤

设计一个新的简单引导组件时，你需要：

| 步骤 | 内容 |
|---|---|
| **① 确定跳转类型** | 这个卡片点击后跳到什么页面？ |
| **② 定义 target 参数** | 前端需要哪些信息来构造目标页 URL？ |
| **③ 设计卡片样式方向** | 这种跳转类型的卡片应该长什么样？（提供设计意图即可，不需要具体 CSS） |
| **④ 编写 AI 触发说明** | 什么对话意图应触发此卡片？AI 如何从对话中提取 target 参数？ |
| **⑤ 提供测试数据** | 1-2 个完整 JSON 示例 |

### 7.5 AI 触发说明要点

简单引导组件的 AI 触发说明与复杂组件略有不同：

| 环节 | 复杂组件 | 简单引导组件 |
|---|---|---|
| **触发时机** | 用户想看数据/图表时 | 用户想跳转到某个页面，或 AI 判断引导用户去某个页面更合适时 |
| **参数提取** | 提取查询参数（时间范围、市场等） | 提取跳转目标（资产 ID、页面类型等） |
| **AI 主动性** | 通常被动——用户明确要求 | 可以主动——AI 判断当前回答不如直接引导用户去看 |

**AI 主动触发示例：**

- 用户问"苹果最新财报怎么样" → AI 可以在文字回答之外，额外附上一个跳转到苹果财报详情页的引导卡片
- 用户问"帮我看看有什么好的投资策略" → AI 可以附上跳转到策略中心的引导卡片

### 7.6 边界情况处理

简单引导组件的边界情况比复杂组件简单，主要集中在**跳转目标异常**：

| 异常场景 | 处理方式 |
|---|---|
| `target.id` 对应的页面不存在 | 前端显示"页面不可用"提示，卡片仍可见但不可点击 |
| `target.type` 不在前端支持的枚举中 | 前端降级为通用样式卡片，跳转到兜底页面 |
| `label` 为空 | 前端使用默认文案"查看详情" |
| 目标页面需要登录 | 前端正常跳转，目标页自行处理登录拦截 |

### 7.7 设计自检 Checklist（简单引导组件）

- [ ] 跳转目标明确——只看 JSON 能说出点击后跳到哪个页面
- [ ] `target.type` 使用了枚举约束
- [ ] AI 触发说明覆盖了主动触发和被动触发两种场景
- [ ] 不同 `target.type` 的卡片样式方向已描述
- [ ] 考虑了目标页面不存在的降级方案
- [ ] 提供了至少 1 个完整 JSON 示例

## 附录 C：术语表

| 术语 | 含义 |
|---|---|
| **业务组件** | AI 传入参数、前端负责后续处理的组件。与通用图表（AI 填充全部数据）相对 |
| **复杂组件** | 业务组件的子类：复用产品已有组件或新设计带复杂交互的卡片，前端调 API 取数据后渲染 |
| **简单引导组件** | 业务组件的子类：单一点击跳转行为，渲染为样式卡片，不同跳转类型可有不同样式 |
| **触发意图** | 用户对话中隐含的、应该激活某个业务组件的意图 |
| **参数提取** | AI 从用户自然语言中识别出组件所需参数值的过程 |
| **枚举（enum）** | 参数的合法取值列表，限制只能从预定义选项中选择 |
| **降级（Fallback）** | 当参数异常或 API 失败时，系统退而求其次的展示方式 |
| **骨架屏** | 组件加载过程中的占位动画，展示组件大致形状但无实际数据 |
| **导航（navigate）** | 简单引导组件的标准行为——将用户从当前对话界面引导到目标页面 |
